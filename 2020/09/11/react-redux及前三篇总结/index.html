react-redux及前三篇总结

9月 11, 2020 发布在 React

解决了随时访问store,又不会出现混乱

文档地址：https://react-redux.js.org/5.x/api

常用的API

  • Provider
  • connect 「连接UI组件和容器组件(store)」

使用方法：

1. 安装react-redux
2. 使用Provider把App包起来，把store传给Provider
3. App获取store，获取初始值

App中调用connect，注意connect的调用方式，它接受两次参数，怎样让一个函数接受两次参数呢？只要让这个函数return出一个函数，这里是一个测试的例子，为什么connect可以调两次，因为他返回了一个新的函数，他会把第一个参数，第二个参数结合起来之后，产生一些魔法

在这里connect(1)(2)，高亮部分是一个偏函数（因为它的参数不全）

4. 实现加1

这时候，需要给connect传另一个参数，它是一个对象，生成一个action

总结一下：

这里使用第三种方式（react-reudx）实现了count中的加1功能，在App中使用connect，不需要获取整个store

  • connect中的参数
    • 第一个参数（mapStateToProps）map：必须是一个函数，接受一个state映射，把全局的state映射到当前当前组件的props,接受一个state,把state变成props的n。App需要获取哪些store的state
    • 第二个参数（mapDispatchToprops）：如何把dispatch映射到props，接受一个dispatch,把dispatch映射到props的add1。 需要生成哪些action。
    • connect把数据(初始状态)和action(状态变更)还有组件组合起来，产生一些魔法
  • 梳理一下从获取n到更新n的过程（不是直接更新自己的n）：
    • 通过this.props.add1(在connect的第二个参数中生产)去触发一个action
    • action会触发一个新的state
    • 新的state通过Provider传到App
    • App再重新获取n，n再重新渲染到页面
三种方式对比

回顾一下三个例子，分别用了js+redux 、react+redux、使用库react-redux实现了count功能

  • 例1
    • 引入redux，生产store，stateChange记录状态变更的过程，根据action-type生产新的state，导致render再运行
    • 三个重点
      • dispatch一个action
      • 根据action生成一个新的state,并触发一个change事件
      • 接受到事件，重新render整个UI
  • 例2
    • 和例1的变化：由js变成了react,好处是每次都是局部更新，其他和例1完全一样
  • 例3
    • 没有把store传给App,而是传给了Provider,好处就是Provider会把store传给它里面的每一个组件，不用一层一层的往下传。然后App通过connect读store,通过this.props.XXX拿到值，当值要改变的时候，会产生一个action，相比于前面两个例子，省掉了：新的state触发APP的重新渲染（state变化了，provider就知道，就会通知所有组件，哪里有改变就去更新）
    • 特点
      • 所有的数据都从props中获取
几个API之间的对应
  • store.getState()
  • event(name, type) -> dispatch(type, payload)
  • trriger -> dispatch (触发)
  • On -> Subscribe（订阅，render之后重新触发）
  • stateChange -> Reducer（根据旧的state生成新的state）
